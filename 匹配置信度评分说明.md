# 匹配置信度评分说明

本文档说明 JSON 报告中 `matches[].score` 和部分 `changes[].details.score` 的含义。评分详情用于调试、审计、UI 展示或交给其他 AI 分析，不作为额外业务决策字段。

## JSON 结构

每个成功匹配的节点都会在 `matches[]` 中带有 `score`：

```json
{
  "confidence": 92,
  "score": {
    "total": 92,
    "reasons": ["same_name", "same_structure_hash"],
    "parts": [
      {"kind": "add", "reason": "same_name", "value": 12}
    ]
  }
}
```

字段含义：

- `confidence`：兼容旧字段，等于 `score.total`。
- `score.total`：最终匹配分数，范围为 `0..100`。
- `score.reasons`：简化原因列表，方便快速筛选。
- `score.parts`：详细评分组成，每一项说明分数如何变化。

`changes[].details.score` 只会出现在匹配风险类变化和节点移动/重命名类变化中。普通字段、资源、事件、组件变化不会重复携带 `score`，可通过 `before_internal_path` / `after_internal_path` 到 `matches[]` 中查找对应评分。

## parts.kind 类型

| kind | 含义 |
| --- | --- |
| `add` | 普通加分项。 |
| `penalty` | 扣分项，`value` 为负数。 |
| `overlap` | 集合重叠加分，如组件、资源、Label 文本、事件集合的交集率。 |
| `identity` | 强身份确认项，目前用于唯一身份哈希相同。 |
| `floor` | 保底修正，把分数提高到指定下限。 |
| `cap` | 封顶修正，把分数限制到指定上限。 |
| `info` | 纯说明标记，不改变分数。 |

`overlap` 项的 `details` 包含：

- `weight`：该项满分权重。
- `intersection`：双方集合交集数量。
- `union`：双方集合并集数量。

`floor` 和 `cap` 项的 `details` 包含：

- `before`：修正前分数。
- `after`：修正后分数。

## 评分维度

总分由多个维度累加，最终封顶为 `100`，最低为 `0`。如果某个维度的整体哈希已经完全相同，就不会再计算该维度下的 overlap 细项。

### 名称

| reason | kind | 分值 | 触发条件 |
| --- | --- | ---: | --- |
| `same_name` | `add` | `+12` | 节点名字完全相同。 |
| `low_info_name` | `penalty` | `-4` | 节点名字不同，且任一方名字是低信息名。 |

当前低信息名来自 `config.py` 的 `LOW_INFO_NODE_NAMES`，包括 `bg`、`icon`、`label`、`title`、`num`、`txt`、`text`、`node`、`con`、`item`、`btn`。

### 结构

| reason | kind | 分值 | 触发条件 |
| --- | --- | ---: | --- |
| `same_structure_hash` | `add` | `+18` | 结构指纹完全相同，主要基于组件类型和子节点名字。 |
| `component_overlap` | `overlap` | `0..18` | 结构指纹不同，但组件类型集合有重叠。 |
| `child_name_overlap` | `overlap` | `0..8` | 结构指纹不同，但子节点名字集合有重叠。 |

### 视觉

| reason | kind | 分值 | 触发条件 |
| --- | --- | ---: | --- |
| `same_visual_hash` | `add` | `+20` | 视觉指纹完全相同，主要基于节点属性、资源 UUID 和 Label 文本。 |
| `resource_overlap` | `overlap` | `0..18` | 视觉指纹不同，但资源 UUID 集合有重叠。 |
| `label_overlap` | `overlap` | `0..14` | 视觉指纹不同，但 Label 文本集合有重叠。 |

### 行为

| reason | kind | 分值 | 触发条件 |
| --- | --- | ---: | --- |
| `same_behavior_hash` | `add` | `+22` | 行为指纹完全相同，主要基于脚本类型和事件绑定。 |
| `script_overlap` | `overlap` | `0..18` | 行为指纹不同，但脚本类型集合有重叠。 |
| `event_overlap` | `overlap` | `0..18` | 行为指纹不同，但事件绑定集合有重叠。 |

### 位置与上下文

| reason | kind | 分值 | 触发条件 |
| --- | --- | ---: | --- |
| `same_parent_path` | `add` | `+8` | 父节点显示路径相同。 |
| `same_path` | `add` | `+8` | 节点显示路径相同。 |
| `same_sibling_index` | `add` | `+4` | 在父节点中的兄弟序号相同。 |
| `matched_parent` | `add` | `+14` | 父节点已经匹配，且当前候选位于对应父节点下。 |
| `different_matched_parent` | `penalty` | `-10` | 父节点已经匹配，但当前候选不在对应父节点下，且内容相似度不够强。 |

`matched_parent` 只会在候选排序阶段出现，取决于父节点是否已经提前匹配。

### 信息量惩罚

| reason | kind | 分值 | 触发条件 |
| --- | --- | ---: | --- |
| `low_information_node_penalty` | `penalty` | `-18` | 双方都没有组件，且双方名字都是低信息名。 |

## 特殊修正项

这些项不是普通维度得分，而是匹配流程中的规则标记或修正。

| reason | kind | 分值 | 含义 |
| --- | --- | ---: | --- |
| `same_internal_path_anchor` | `info` | `0` | 通过 internal path 锚点匹配。internal path 是带同名兄弟索引的路径。 |
| `same_internal_path_anchor_floor` | `floor` | `74` | internal path 锚点安全匹配时，分数至少为 `PROBABLE_SCORE`。 |
| `same_path_anchor` | `info` | `0` | 通过显示路径锚点匹配。 |
| `same_path_anchor_floor` | `floor` | `74` | 显示路径锚点安全匹配时，分数至少为 `PROBABLE_SCORE`。 |
| `same_unique_identity_hash` | `identity` | `100` | 唯一身份哈希相同，直接确认匹配。 |
| `duplicate_identity` | `info` | `0` | 身份哈希相同，但该身份在至少一侧存在重复。 |
| `duplicate_identity_cap` | `cap` | `91` | 重复身份场景不能直接满分确认，最高限制为 `CONFIRMED_SCORE - 1`。 |

## 匹配状态阈值

| 分数 | status | 含义 |
| ---: | --- | --- |
| `>= 92` | `confirmed` | 确认匹配。 |
| `>= 74` | `probable` | 较可信匹配。 |
| `>= 58` | `uncertain` | 不确定匹配。 |
| `< 58` | `unmatched` | 不匹配。 |

如果最佳候选和第二候选的分差小于等于 `4`，会优先标记为 `ambiguous`。这条规则优先于上面的分数阈值，因此高分匹配也可能因为候选过近而被标记为模糊匹配。

颜色不属于 JSON 结构的一部分，可由 UI 根据 `status` 或 `confidence` 自行映射。

📖 置信度说明

置信度表示工具判断「before 和 after 中两个节点是否为同一个节点」的量化把握程度（0 ~ 100）。

分数不是「相似度」，而是综合了节点名称、组件结构、视觉属性、行为特征和上下文位置后的加权评分。

评分维度

维度匹配条件加分
名称节点名相同+12
名称低信息名（bg/label/icon 等）-4
结构structure_hash 完全相同+18
结构组件类型重叠（Jaccard）最多 +18
结构子节点名称重叠（Jaccard）最多 +8
视觉visual_hash 完全相同+20
视觉资源 uuid 重叠（Jaccard）最多 +18
视觉Label 文本重叠（Jaccard）最多 +14
行为behavior_hash 完全相同+22
行为脚本类型重叠（Jaccard）最多 +18
行为事件 handler 重叠（Jaccard）最多 +18
上下文父路径相同+8
上下文完整路径相同+8
上下文兄弟下标相同+4
惩罚无组件 + 低信息名称-18

分级标准

分数区间状态含义
>= 92confirmed高置信匹配，直接视为同一节点进入 diff
74 ~ 91probable大概率是同一节点，进入 diff，报告中展示置信度
58 ~ 73uncertain不确定，不自动视为同一节点，进入低置信度列表
< 58unmatched视为新增或删除

多候选（Ambiguous）

当最高分与第二名差距 <= 4 分时，即使总分达到 confirmed/probable，也会被标记为 ambiguous（多候选）。

这表示工具发现了多个"equally likely"的匹配对象，必须由人工判断哪个才是真正的对应节点。

低信息节点

名称属于以下名单的节点被视为低信息节点，名称权重会被降低：

bg, icon, label, title, num, txt, text, node, con, item, btn

这些节点通常数量多、特征少，容易互相错配。工具会提高资源 uuid、组件结构和父子关系在评分中的权重来辅助消歧。

为什么需要置信度？

普通文本 diff 只看路径，会把「移动节点」误报为「删除 + 新增」。PfbDiff 用指纹匹配来识别同一节点，但如果没有置信度机制，当两个 prefab 存在大量相似节点时（如列表中的多个 Item），很容易把 A 错配成 B，导致真正的变化被掩盖。

置信度机制把这种风险量化出来：不敢确定的就告诉用户，让用户做最终判断。

低置信度在报告中如何识别？

在树形对比报告中，低置信度节点有以下特征：

  • 节点行左侧边框为 深红色（#dc2626）
  • Badge 标签显示「低置信度」或「多候选」
  • 点击节点后，详情面板显示 置信度: 62 之类的分数

低置信度的风险分级

节点特征风险等级说明
低置信度 + 包含脚本或事件高风险错配可能掩盖业务逻辑变化，必须人工确认
低置信度 + 纯 UI 展示节点中风险视觉可能错位，但不会影响代码逻辑

看到低置信度时应该怎么做？

  1. 点击节点查看详情面板，看置信度分数和匹配依据（如 same_name、resource_overlap）
  2. 对比左右两栏该节点的路径、组件、资源，判断工具配得对不对
  3. 如果配错了：在人工合并时忽略工具的匹配结论，按你自己的判断处理
  4. 如果配对了：可以忽略这个警告，继续审查其他变化

一句话总结

低置信度 = 工具在「举手投降」。它发现了一些相似特征，但不敢打包票说这是同一个节点。深红色就是它的求救信号——请你来做最终判断，避免错配导致真正变化被掩盖。
